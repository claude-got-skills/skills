```python
        "file_id": file_response.id,
        "extraction": extracted,
        "enriched": enriched,
        "summary": summary_response.content[0].text

def enrich_with_kb(extracted: dict, kb_lookup_fn) -> dict:
    """Lightweight KB enrichment."""
    for entity in extracted.get("entities", []):

            entity["kb_id"] = match["id"]
            entity["kb_summary"] = match["summary"]
    return extracted
```

```python
    key: str,
```

```python
    default: Any = None,
    missing_key_behavior: str = "skip"
) -> List[Dict[str, Any]]:
    """
    Sort a list of dictionaries by a specified key.
    
    Args:
        items: List of dictionaries to sort
        key: The dictionary key to sort by
        reverse: If True, sort in descending order (default: False)
        default: Default value for missing keys (default: None)

            - "skip": Items with missing keys are placed at the end

        KeyError: If missing_key_behavior is "error" and key is missing
        ValueError: If missing_key_behavior is invalid

        [{'name': 'Bob', 'age': 25}, {'name': 'Alice', 'age': 30}]

    if missing_key_behavior not in ("skip", "default", "error"):
        raise ValueError(
            f"invalid missing_key_behavior: {missing_key_behavior}"

        # Validate all items have the key
        for item in items:
            if key not in item:
                raise KeyError(f"Key '{key}' not found in item: {item}")
        return sorted(items, key=lambda x: x[key], reverse=reverse)

        # Use default value for missing keys
        return sorted(
            items,
            key=lambda x: x.get(key, default),
            reverse=reverse

        # Separate items with and without the key
        with_key = [item for item in items if key in item]
        without_key = [item for item in items if key not in item]

        sorted_with_key = sorted(
            with_key,
            key=lambda x: x[key],
            reverse=reverse
        )
        
        # Append items without the key at the end
        return sorted_with_key + without_key
```

# TCP vs UDP: Key Differences

## Overview

TCP (Transmission Control Protocol) and UDP (User Datagram Protocol) are the two primary transport layer protocols in the Internet Protocol Suite. They sit at Layer 4 (Transport Layer) of the OSI model and determine how data is sent between applications.

## Side-by-Side Comparison

| Feature | TCP | UDP |
|---------|-----|-----|
| **Connection** | Connection-oriented | Connectionless |
| **Reliability** | Guaranteed delivery, in-order | Best-effort, no guarantees |
| **Ordering** | Maintains packet order | No guaranteed order |
| **Speed** | Slower (more overhead) | Faster (minimal overhead) |
| **Error Checking** | Extensive | Basic checksum only |
| **Flow Control** | Yes (prevents overwhelming receiver) | No |
| **Congestion Control** | Yes | No |
| **Header Size** | 20 bytes (minimum) | 8 bytes |
| **Broadcasting** | No | Yes |

## Detailed Differences

### **TCP: Reliability First**
- **Connection setup**: Requires a handshake (SYN, SYN-ACK, ACK) before data transfer
- **Acknowledgments**: Each packet is confirmed received
- **Retransmission**: Lost packets are resent automatically
- **Sequencing**: Packets are numbered and reassembled in correct order
- **Flow control**: Uses windowing to prevent overwhelming the receiver

### **UDP: Speed First**
- **No setup**: Data is sent immediately without connection establishment
- **Fire-and-forget**: No confirmation of delivery
- **No retransmission**: Lost packets are simply lost
- **No ordering**: Packets can arrive out of sequence (application must handle)
- **Minimal overhead**: Just a checksum for basic error detection

## When to Use Each

### **Use TCP When:**

1. **Data integrity is critical**
   - File transfers (FTP, SFTP)
   - Email (SMTP, POP3, IMAP)
   - Web browsing (HTTP/HTTPS)
   - Database operations

2. **Order matters**
   - Completing a sequence of operations that depend on each other
   - Financial transactions

3. **You need automatic error recovery**
   - You don't want to implement retry logic yourself

4. **Bandwidth isn't extremely constrained**
   - The overhead of acknowledgments and flow control is acceptable

### **Use UDP When:**

1. **Speed is more important than perfection**
   - Online gaming (real-time multiplayer)
   - Voice/video calls (VoIP, video conferencing)
   - Live streaming
   - Occasional lost frames are acceptable; late delivery is worse

2. **Broadcasting or multicasting**
   - Sending data to multiple recipients simultaneously
   - Network announcements
   - Monitoring/alerting systems

3. **Low-latency communication is essential**
   - DNS queries (though they can also use TCP)
   - IoT sensor updates (occasional loss acceptable)
   - Stock ticker updates

4. **Application can handle retries**
   - DNS: Client retransmits if no response
   - Game servers: Often implement custom retry logic
   - You have application-level error handling

5. **Bandwidth is extremely limited**
   - Minimal protocol overhead matters
   - Embedded devices on constrained networks

## Practical Examples

**TCP Examples:**

```
Email Client → Mail Server (SMTP)
Git Push/Pull → Repository
SSH Terminal Session
```

**UDP Examples:**
```
Skype/Discord Voice Call
Online Multiplayer Game
Netflix Stream
DNS Lookup
NTP Time Sync
```

## Hybrid Approach

Modern applications sometimes use **both**:
- **QUIC** (Quick UDP Internet Connections): Implements TCP-like reliability on top of UDP for faster, more flexible connections
- **Game servers**: Often use UDP for real-time gameplay but TCP for critical operations (login, payment)
- **Streaming services**: UDP for video with TCP for control/metadata

## Summary

Choose **TCP** when you can't afford to lose data and can tolerate some latency. Choose **UDP** when speed matters more than perfect delivery and you can tolerate occasional data loss. In practice, TCP dominates for mission-critical services, while UDP powers everything where real-time performance is paramount.

# Preventing Unnecessary Re-renders in React

You have several options depending on your situation:

## 1. **React.memo** (Simplest for most cases)

Wrap your component to skip re-renders when props haven't changed:

```jsx
const MyComponent = React.memo(({ name, age }) => {
  return <div>{name} is {age}</div>;
});
```

This does a **shallow comparison** of props. If props are the same, the component won't re-render.

**When to use**: When your component receives primitive values (strings, numbers) or stable object references.

## 2. **useMemo** (For expensive computations)

If your component does heavy calculations, memoize the result:

```jsx
const MyComponent = ({ data }) => {
  const expensiveResult = useMemo(() => {
    return complexCalculation(data);
  }, [data]); // Only recalculates when `data` changes

  return <div>{expensiveResult}</div>;
};
```

## 3. **useCallback** (For function props)

If you pass functions to child components, wrap them to prevent recreating on every render:

```jsx
const Parent = () => {
  const handleClick = useCallback(() => {
    console.log('clicked');
  }, []); // Empty deps = stable reference

  return <Child onClick={handleClick} />;
};

const Child = React.memo(({ onClick }) => (
  <button onClick={onClick}>Click me</button>
));
```

Without `useCallback`, the function reference changes every render, breaking `React.memo`.

## 4. **Identify unstable object/array props**

This is a common culprit:

```jsx
// ❌ BAD: New object created on every render
<MyComponent config={{ color: 'red' }} />

// ✅ GOOD: Move outside or memoize
const config = { color: 'red' };
<MyComponent config={config} />

// ✅ GOOD: Or use useMemo
const config = useMemo(() => ({ color: 'red' }), []);
<MyComponent config={config} />
```

## 5. **Check your dependency arrays**

Incorrect dependencies can cause unnecessary re-renders:

```jsx
// ❌ BAD: Re-runs on every render
useEffect(() => { ... }, [])

// ✅ GOOD: Only when specific values change
```
