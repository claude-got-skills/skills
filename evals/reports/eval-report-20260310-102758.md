# Simple Document Analysis Pipeline

Here's a pragmatic architecture for your constraints:

## Core Components

**1. Document Processing Layer**
```
PDF Input → Text Extraction → Chunking → Storage
```
- Use **PyPDF2** or **pdfplumber** for extraction (reliable, lightweight)
- Chunk into ~1000-token segments with overlap for context
- Store in a simple PostgreSQL table with metadata (doc_id, page_num, text)

**2. Extraction & Cross-Reference**
```
Chunks → LLM extraction → Knowledge base lookup → Deduplication
```
- Call an LLM (OpenAI API or local Llama2) with a focused prompt per chunk
- Extract: names, dates, amounts, entities relevant to your domain
- Query your KB via SQL/vector similarity (depending on complexity)
- Simple dedup: hash extracted values, store in results table

**3. Aggregation & Summary**
```
Extracted data → Synthesis → Structured output (JSON)
```
- Combine results across chunks, handle conflicts (majority vote or recency)
- One final LLM call to synthesize into a clean summary
- Return structured JSON with confidence scores

## Minimal Stack

```
FastAPI (REST endpoints)
├── PostgreSQL (documents, chunks, results)
├── LLM API or local model
├── Redis (optional: caching, task queue)
└── Simple file storage (S3 or local filesystem)
```

## Key Decisions for "Small Team"

| Choice | Why |
|--------|-----|
| **Async processing** | Queue jobs, don't block user requests |
| **Batch extraction** | Process chunks in parallel, not sequentially |
| **Cached KB lookups** | Hit the same KB entries repeatedly |
| **Simple prompts** | Easier to debug, cheaper tokens, faster |

## Sample Flow

```python
# Pseudo-code
1. Upload PDF → extract_text() → save chunks to DB
2. For each chunk, call: extract_entities(chunk, domain_context)
3. For each entity, query: kb.lookup(entity_type, value)
4. Aggregate results → synthesize_summary()
5. Return JSON: {entities, cross_refs, summary, confidence}
```

## Avoid Overengineering

❌ Don't build a custom vector DB yet — use PostgreSQL + pgvector if you need similarity search later
❌ Don't fine-tune models — use prompt engineering first
❌ Don't build a UI — start with a CLI or Postman

## What to Watch

- **Token costs**: Set a per-document budget; if a PDF costs $5 to process, rethink your strategy
- **LLM hallucinations**: Always validate extracted data against KB; flag uncertain matches
- **Chunk boundaries**: Breaking mid-sentence ruins extraction—use sentence-aware chunking

**Start here, iterate once you have real data.** Most complexity emerges from domain-specific edge cases, not architecture.

# Recommended Architecture for Document Analysis Pipeline

For a small team processing 100-page PDFs, here's a practical approach that balances simplicity with capability:

## Core Architecture

**Single-pass analysis with structured outputs:**

1. **Upload PDF via Files API** → store file_id for reuse
2. **Claude Sonnet 4.6** (200K context) → analyze full PDF in one call
3. **Structured JSON output** → guaranteed schema for downstream use
4. **Optional knowledge base lookup** → lightweight integration if needed

This works because:
- 100-page PDFs fit comfortably in 200K context (PDFs typically use ~3-5 tokens/page, so ~300-500 tokens)
- Single pass = no chunking complexity, no context loss between sections, no orchestration overhead
- Files API enables reuse (avoid re-uploading the same PDF)
- Sonnet 4.6 is cost-effective for this workload

## Concrete Implementation

```python
import anthropic
import json

client = anthropic.Anthropic()

# 1. Upload PDF once, reuse file_id
def upload_pdf(file_path: str) -> str:
    with open(file_path, "rb") as f:
        response = client.beta.files.upload(
            file=(file_path.split("/")[-1], f, "application/pdf"),
        )
    return response.id

# 2. Analyze with structured output
def analyze_document(file_id: str, knowledge_base: dict = None) -> dict:
    kb_context = ""
    if knowledge_base:
        kb_context = f"""
You have access to this knowledge base:
{json.dumps(knowledge_base, indent=2)}

Cross-reference extracted information against it where relevant.
"""
    
    response = client.beta.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=4096,
        messages=[
            {
                "role": "user",
                "content": [
                    {
                        "type": "text",
                        "text": f"""Analyze this document and extract key information.

{kb_context}

Return a structured JSON with:
- summary: 2-3 sentence overview
- key_facts: list of important findings
- entities: names, dates, organizations mentioned
- cross_references: how it relates to known entities in the KB
- confidence: overall confidence level (high/medium/low)
- action_items: any next steps or decisions needed"""
                    },
                    {
                        "type": "document",
                        "source": {
                            "type": "file",
                            "file_id": file_id
                        }
                    }
                ]
            }
        ],
        output_config={
            "type": "json_schema",
            "schema": {
                "type": "object",
                "properties": {
                    "summary": {"type": "string"},
                    "key_facts": {
                        "type": "array",
                        "items": {"type": "string"}
                    },
                    "entities": {
                        "type": "object",
                        "properties": {
                            "people": {"type": "array", "items": {"type": "string"}},
                            "organizations": {"type": "array", "items": {"type": "string"}},
                            "dates": {"type": "array", "items": {"type": "string"}}
                        }
                    },
                    "cross_references": {
                        "type": "array",
                        "items": {
                            "type": "object",
                            "properties": {
                                "entity": {"type": "string"},
                                "kb_match": {"type": "string"},
                                "relevance": {"type": "string"}
                            }
                        }
                    },
                    "confidence": {"type": "string"},
                    "action_items": {"type": "array", "items": {"type": "string"}}
                },
                "required": ["summary", "key_facts", "entities"]
            }
        },
        betas=["files-api-2025-04-14"]
    )
    
    return json.loads(response.content[0].text)

# 3. Process pipeline
if __name__ == "__main__":
```

## Full Implementation Pattern

```typescript
async function reviewContent(
  content: string,
  options: { tier: "fast" | "standard" | "thorough" } = { tier: "standard" }
): Promise<QAResult> {
  // Tier-based routing
  if (options.tier === "fast") {
    // Quick filter only (for time-sensitive content)
    const result = await quickFilter(content);
    return result.includes("PASS") ? { status: "approved" } : { status: "needs_revision" };
  }

  if (options.tier === "standard") {
    // Filter + structured compliance
    const filterResult = await quickFilter(content);
    
    if (filterResult.includes("PASS")) {
      return { status: "approved", confidence: "high" };
    }
    
    // Deep review on flagged content
    return await complianceCheck(content);
  }

  if (options.tier === "thorough") {
    // All three stages
    const filter = await quickFilter(content);
    const deep = await deepReview(content);
    const compliance = await complianceCheck(content);
    
    return {
      status: compliance.status,
      filterIssues: filter,
      deepAnalysis: deep,
      compliance: compliance,
      confidence: "very_high"
    };
  }
}
```

# TCP vs UDP: Key Differences

## Core Differences

| Feature | TCP | UDP |
|---------|-----|-----|
| **Connection** | Connection-oriented (handshake required) | Connectionless (no setup) |
| **Reliability** | Guaranteed delivery, in-order | Best-effort, may lose packets |
| **Ordering** | Maintains packet sequence | No guarantees on order |
| **Speed** | Slower (overhead from reliability) | Faster (minimal overhead) |
| **Header Size** | 20 bytes | 8 bytes |
| **Flow Control** | Yes (prevents overwhelming receiver) | No |
| **Error Checking** | Extensive | Basic checksum only |
| **Broadcasting** | No (point-to-point only) | Yes (can broadcast/multicast) |

## How They Work

**TCP (Transmission Control Protocol)**
- Establishes a connection via a 3-way handshake (SYN, SYN-ACK, ACK)
- Tracks every packet, retransmits lost ones, and reorders arrivals
- Creates a reliable "stream" — like a phone call
- Closes with a formal handshake

**UDP (User Datagram Protocol)**
- Sends packets immediately with no setup
- No acknowledgment of receipt — just sends and hopes
- Simpler and faster — like shouting across a room

## When to Use Each

### Use **TCP** when:
- **Accuracy is critical** — file transfers, email, banking, database queries
- **Order matters** — web browsing (HTTP/HTTPS), SSH, FTP
- **Retransmission is acceptable** — the delay is worth reliability
- **Examples**: Web servers (HTTP), email (SMTP), file transfer (FTP), secure shell (SSH)

### Use **UDP** when:
- **Speed matters more than perfection** — losing a few packets is acceptable
- **Low latency is essential** — live communication can't tolerate retransmission delays
- **Volume is high** — broadcasting to many recipients (multicast)
- **Examples**: 
  - Video/audio streaming (YouTube, Zoom, Discord)
  - Online gaming (real-time multiplayer)
  - DNS queries
  - VoIP (voice calls)
  - Live broadcasts
  - IoT sensor data (occasional loss acceptable)

## Real-World Analogy

**TCP** = Sending a letter with tracking: expensive, slow, but guaranteed to arrive intact and in order.

**UDP** = Shouting a message: fast and cheap, but if there's noise, the recipient might miss part of it.

## Practical Example

A **video call** uses both:
- **UDP** for audio/video streams (occasional glitches acceptable; speed critical)
- **TCP** for signaling/control (setting up the call, reliability important)
