# Simple Document Analysis Pipeline

Here's a pragmatic architecture for a small team:

## Core Components

**1. Document Processing Layer**
```
PDF → Text Extraction → Chunking → Storage
```
- Use **PyPDF2** or **pdfplumber** for extraction (handles most PDFs)
- Split into ~500-1000 token chunks (overlap slightly for context)
- Store chunks with page references in a simple database (PostgreSQL or SQLite)

**2. Information Extraction**
- Run each chunk through an LLM (GPT-4, Claude, or local Llama if budget-conscious)
- Prompt it to extract specific fields: `["dates", "parties", "amounts", "decisions"]`
- Fallback to regex for structured data if high confidence needed

**3. Knowledge Base Cross-Reference**
- Store your knowledge base in the same DB as PDFs (simple embedding index with pgvector or similar)
- Query by semantic similarity or keyword match
- A basic vector DB like **Chroma** or **Weaviate** (self-hosted) keeps overhead down

**4. Summary Generation**
- Feed extracted info + relevant KB matches back to LLM
- Structured prompt to produce consistent JSON output
- Validate schema before storing

## Recommended Tech Stack

```
FastAPI (lightweight API)
    ↓
PostgreSQL + pgvector (storage + embeddings)
    ↓
LangChain or plain OpenAI SDK (orchestration)
    ↓
Pydantic (output validation)
```

## Simple Workflow Example

```python
from langchain.document_loaders import PDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import PgVector

# Load & chunk

docs = loader.load()
splitter = RecursiveCharacterTextSplitter(chunk_size=1000)
```

## Single-Pass Architecture (Recommended)

**Use Claude's native PDF handling + 1M context + structured outputs:**

```python
from anthropic import Anthropic

client = Anthropic()

def analyze_pdf(pdf_base64: str, knowledge_base: str) -> dict:
    """Process a 100-page PDF in a single pass."""
    
    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=16000,
        betas=["context-1m-2025-08-07"],
        output_config={
            "type": "json_schema",
            "schema": {
                "type": "object",
                "properties": {
                    "key_findings": {"type": "array", "items": {"type": "string"}},
                    "entities": {
                        "type": "object",
                        "properties": {
                            "people": {"type": "array"},
                            "organizations": {"type": "array"},
                            "dates": {"type": "array"}
                        }
                    },
                    "knowledge_base_matches": {
                        "type": "array",
                        "items": {
                            "type": "object",
                            "properties": {
                                "reference": {"type": "string"},
                                "relevance": {"type": "string"}
                            }
                        }
                    },
                    "summary": {"type": "string"},
                    "confidence": {"type": "number", "minimum": 0, "maximum": 1}
                },
                "required": ["key_findings", "summary"]
            }
        },
        messages=[
            {
                "role": "user",
                "content": [
                    {
                        "type": "text",
                        "text": f"""Analyze this document thoroughly.

KNOWLEDGE BASE (for cross-reference):
{knowledge_base}

Extract:
1. Key findings and insights
2. Important entities (people, organizations, dates)
3. Matches to the knowledge base above
4. A structured summary

Provide confidence scores for major claims."""
                    },
                    {
                        "type": "document",
                        "source": {
                            "type": "base64",
                            "media_type": "application/pdf",
                            "data": pdf_base64
                        }
                    }
                ]
            }
        ]
    )
    
    return response.content[0].parsed
```

**Why this works:**
- **1M context** handles most 100-page PDFs in one pass (typically 150–400K tokens)
- **Structured output** guarantees valid JSON without parsing
- **Visual layout analysis** captures formatting Claude might miss as text
- **No round-trips** — faster and cheaper than multi-step pipelines
- **Small team overhead** — just one prompt, one API call

# QA System Architecture for AI Content

Here's a practical layered approach:

## 1. **Multi-Stage Pipeline**

```
Generated Content
    ↓
[Rule-Based Checks] → [Model-Based Checks] → [Human Review Queue]
    ↓                      ↓                         ↓
  Fast/Cheap          Nuanced/Flexible         High Stakes
```

**Rule-based layer** (cheap, deterministic):
- Regex/pattern matching for compliance issues
- Length/format validation
- Brand keyword checks
- PII detection

**Model-based layer** (more sophisticated):
- Fine-tuned classifier for tone adherence
- Semantic similarity checks against style examples
- Factual consistency scoring
- Readability metrics

**Human review**: Flag high-risk or borderline cases

## 2. **Key Components**

```
Content Input
    ↓
├─ Metadata Extractor (source, intended use, audience)
├─ Chunker (break long content into reviewable units)
├─ Quality Scorers (parallel checkers):
│  ├─ Accuracy module
│  ├─ Tone/voice module
│  ├─ Style compliance module
│  └─ Custom domain checks
├─ Aggregator (combine signals → confidence score)
└─ Routing Logic:
   ├─ Score >0.9 → Auto-approve
   ├─ 0.7-0.9 → Flag specific issues, suggest fixes
   ├─ <0.7 → Human review required
```

## 3. **Accuracy Checking** (hardest part)

Start honest about limitations:
- **Factual claims**: Cross-reference against trusted sources/databases if available
- **Statistical claims**: Flag for manual verification
- **Domain-specific**: Use domain classifiers to route to specialists
- **Admit uncertainty**: "Cannot verify automatically" is valid output

Consider:
- RAG systems if you have reliable knowledge bases
- Integrating with fact-checking APIs
- Building confidence thresholds, not binary answers

## 4. **Tone & Style**

- Train a classifier on approved/rejected examples
- Use reference embeddings of your brand voice
- Semantic distance to style guide examples
- Allow configurable strictness per content type

## 5. **Practical Implementation**

**Tech stack considerations:**
- Queue system (Redis/Celery) for routing to human review
- Scoring: lightweight models (distilled BERT variants)
- Storage: version all outputs with scores + human feedback
- Feedback loop: rejected content trains next iteration

**Critical**: Build feedback mechanisms
- Track which auto-approvals customers complain about
- Use human review decisions to retrain models
- Monitor false positive rates (you'll reject good content)

## 6. **Real Constraints**

**Know what you can't automate:**
- Subtle contextual appropriateness often needs humans
- Domain expertise gaps (medical, legal, financial content)
- Cultural nuance and sensitivity
- Novel false claims you haven't seen before

**Start conservative**: Begin with manual review of everything, use ML to surface issues and speed human review. Gradually increase automation as you validate.

**Measure:**
- False positive rate (wrongly rejected good content)
- False negative rate (let bad content through)
- Time saved vs. quality lost

# QA System for AI-Generated Content

Here's a practical architecture that scales with your content volume and quality needs:

## Core Architecture Decision

Use **subagents** (if Claude Code/Agent SDK) or **parallel tool calls** (if API-only). This lets you:
- Run independent checks (accuracy, tone, compliance) without serial bottlenecks
- Apply different models/effort levels per check type
- Isolate failures without affecting the whole pipeline

## Recommended Setup

```
Input Content
    ↓
[Orchestrator Agent]
    ↓
    ├─→ [Accuracy Checker] — fact verification, hallucination detection
    ├─→ [Tone Analyzer] — brand voice alignment, sentiment check
    └─→ [Compliance Checker] — style guide rules, format validation
    ↓
[Result Aggregator] — merge verdicts, assign severity
    ↓
Output (Pass/Flag for Review/Reject)
```

## Implementation Patterns

```typescript
  const [response, setResponse] = useState("");
  const [loading, setLoading] = useState(false);

  const sendMessage = useCallback(
    async (messages: any[], userContext: string) => {
      setLoading(true);
      setResponse("");

      try {
        const res = await fetch("/api/chat", {
          method: "POST",
          headers: { "Content-Type": "application/json" },
          body: JSON.stringify({ messages, userContext }),
        });

        if (!res.ok) throw new Error("API request failed");

        const reader = res.body?.getReader();
        const decoder = new TextDecoder();

        while (true) {
          const { done, value } = await reader!.read();
          if (done) break;

          const chunk = decoder.decode(value);
          const lines = chunk.split("\n");

          for (const line of lines) {
            if (line.startsWith("data: ")) {
              const data = line.slice(6);
              if (data === "[DONE]") break;

              try {
                const parsed = JSON.parse(data);
                setResponse((prev) => prev + parsed.token);
              } catch {
                // Skip invalid JSON
              }
            }
          }
        }
      } catch (error) {
        console.error("Streaming error:", error);
      } finally {
        setLoading(false);
      }
    },
    []
  );
```

# Configuring Thinking Effort on Claude Opus 4.6

Great question — Opus 4.6 gives you fine-grained control over reasoning depth via **adaptive thinking** and the **effort parameter**.

## Two Control Levers

### 1. **Adaptive Thinking** (Primary)
Enable extended reasoning that scales automatically:

```json
{

  "messages": [...]
}
```

Adaptive thinking automatically allocates thinking tokens based on problem complexity. Good for mixed workloads where you don't know upfront which tasks need heavy reasoning.

Fine-tune how hard Claude thinks on a specific request:

```json
{
  "model": "claude-opus-4-6-20250514",
  "max_tokens": 16000,
  "thinking": {
    "type": "adaptive"
  },
  "effort": "low",  // or "medium" (default), "high", "max"
  "messages": [...]
}
```

**Effort levels:**
- **`"low"`** — Minimal thinking; fast responses for simple tasks (lookups, formatting)
- **`"medium"`** — Balanced reasoning (default when effort not specified)
- **`"high"`** — Deep reasoning for complex problems
- **`"max"`** — Maximum thinking tokens (Opus 4.6 only; uses 128K output)

```python
def call_claude(task_type: str, prompt: str):

        "lookup": "low",           # "What's the capital of France?"
        "summary": "medium",       # "Summarize this article"
        "analysis": "high",        # "Compare these two approaches"
        "breakthrough": "max",     # "Design a novel algorithm"

        model="claude-opus-4-6-20250514",
        max_tokens=16000,

            "type": "adaptive"
        },

            "role": "user",
            "content": prompt
```

# Automate Linting & Type Checks on File Save

Yes! This is a perfect use case for a **Hook** in Claude Code. Hooks run deterministically on lifecycle events (like file edits) without consuming tokens, making them ideal for this workflow.

## Quick Solution: Pre-Save Hook

Use a **PreSave hook** to automatically run linting and type checks whenever you edit a Python file:

```typescript
// In your Claude Code project
export const hooks = {
  preSave: async (file) => {
    if (!file.path.endsWith('.py')) return; // Only Python files

    const { exec } = require('child_process');
    const path = require('path');

    const dir = path.dirname(file.path);
    const results = {
      linting: null,
      typeCheck: null,
      errors: []
    };

    // Run linting (e.g., ruff or flake8)
    try {
      await new Promise((resolve, reject) => {
        exec(`ruff check "${file.path}" --fix`, { cwd: dir }, (err, stdout, stderr) => {
          if (err) results.errors.push(`Linting failed: ${stderr}`);
          else results.linting = 'passed';
          resolve();
        });
      });
    } catch (e) {
      results.errors.push(`Linting error: ${e.message}`);
    }

    // Run type check (e.g., mypy)
    try {
      await new Promise((resolve, reject) => {
        exec(`mypy "${file.path}"`, { cwd: dir }, (err, stdout, stderr) => {
          if (err) results.errors.push(`Type check failed: ${stderr}`);
          else results.typeCheck = 'passed';
          resolve();
        });
      });
    } catch (e) {
      results.errors.push(`Type check error: ${e.message}`);
    }

    // Optionally notify in Claude Code console
    if (results.errors.length > 0) {
      console.warn('⚠️ Quality checks:', results);
    } else {
      console.log('✅ All checks passed');
    }
  }
};
```

| Feature | Benefit |
|---------|---------|
| **No tokens consumed** | Runs deterministically, doesn't involve Claude |
| **Automatic** | Triggers on save without you remembering |
| **Fast** | Runs locally, no API latency |
| **Deterministic** | Same checks every time, no variance |

```typescript
};
```
