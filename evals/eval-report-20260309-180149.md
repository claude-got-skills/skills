# Simple Document Analysis Pipeline

Here's a practical approach for a small team:

## Architecture Overview

```
PDF Input → Text Extraction → Chunking → Analysis → Storage → Output
              ↓
        Knowledge Base (Vector DB)
```

## Recommended Stack

**Processing Layer:**
- **PyPDF2** or **pdfplumber** for extraction (pdfplumber is more reliable for structured data)
- **LangChain** or **LlamaIndex** to orchestrate the pipeline
- **OpenAI API** or **Claude API** for analysis (simpler than self-hosted LLMs initially)

**Knowledge Base:**
- **Pinecone** or **Supabase (pgvector)** for vector embeddings
- Start with Supabase if you want to avoid another vendor; it's cheaper at small scale

**Storage:**
- **PostgreSQL** for structured metadata
- Same database can store your processed summaries

## Simple Implementation Pattern

```python
from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Supabase
from langchain.chains import RetrievalQA
from langchain.llms import OpenAI

# 1. Load and split
loader = PyPDFLoader("document.pdf")
documents = loader.load()
splitter = RecursiveCharacterTextSplitter(chunk_size=1000, overlap=100)
chunks = splitter.split_documents(documents)

# 2. Store in knowledge base
embeddings = OpenAIEmbeddings()
vectorstore = Supabase.from_documents(chunks, embeddings, client=supabase_client)

# 3. Query with context
qa_chain = RetrievalQA.from_chain_type(
    llm=OpenAI(),
    retriever=vectorstore.as_retriever(),
    return_source_documents=True
)

result = qa_chain("Extract key financial metrics and cross-reference with industry standards")
```

## Key Decisions

| Choice | Reasoning |
|--------|-----------|
| **API vs. local LLM** | Start with API (OpenAI/Claude). Switching later is easy; complexity isn't worth it yet. |
| **Chunking strategy** | 1000-token chunks with 100-token overlap works well for most documents. Adjust based on your domain. |
| **Vector DB** | Supabase is cheapest for small teams. Pinecone if you need managed scaling. |
| **Async processing** | Add Celery/RQ only if you're processing >50 docs/day. |

## What to Avoid Early

- Don't build a custom vector DB
- Don't fine-tune models before you have 500+ examples
- Don't over-engineer error handling initially—use try/except + logging

## Scaling Checkpoints

When you hit these, reconsider:
- **>500 PDFs/month** → Add job queue (Celery/BullMQ)
- **Latency matters** → Switch to local embeddings model
- **Cost spikes** → Consider open-source LLM (Llama 2) or batch processing

Here's a practical architecture that scales from prototype to production:

## Overall Pattern

Use **subagents** for parallel, focused review (accuracy, tone, compliance) → **synthesis agent** for final decisions. This isolates concerns, allows independent tuning, and keeps token usage efficient.

```
Input Content
    ↓
[Accuracy Subagent] → Fact check, references
[Tone Subagent]     → Voice match, sentiment
[Compliance Agent]  → Style guide adherence
    ↓
[Synthesis Agent]   → Aggregate, decide: PASS/FAIL/REVISE
    ↓
Output: {status, issues, suggestions}
```

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def qa_pipeline(content: str, knowledge_base: str = None) -> dict:

            executor.submit(review_tone, content): "tone",
            executor.submit(review_compliance, content): "compliance"

        results = {}
        for future in as_completed(futures):
            key = futures[future]
            try:
                results[key] = future.result()
            except Exception as e:
                results[key] = {"error": str(e), "requires_revision": True}
    
    # Synthesis step
    return synthesize_results(content, results)
```

# Internet Access for Claude

Claude **cannot directly access the internet or make HTTP requests by default**. The API itself has no built-in capability to initiate outbound connections to external services.

However, there are several ways to get Claude live data from your REST API:

## 1. **Code Execution (Recommended for Testing)**
Claude can execute Python code, and that code *can* make HTTP requests:

```python
import requests

response = requests.get('https://your-api.com/endpoint')
data = response.json()
print(data)
```

This works if your API is publicly accessible. However, **code execution runs in a sandboxed container with limited egress** — it can reach public internet but not private networks.

**When to use:** Development, public APIs, proof-of-concept.

## 2. **Web Search / Web Fetch Tools (GA)**
For public data, you can enable web tools:

```json
{
  "tools": [
    {"type": "web_search_20250305", "name": "web_search"},
    {"type": "web_fetch_20260209", "name": "web_fetch"}
  ]
}
```

Claude can fetch and parse HTTP responses. This is simpler than code execution for structured data.

**When to use:** Public REST endpoints, no authentication needed.

## 3. **MCP Connector (Best for Production)**
Use the **MCP Connector** (beta, header `mcp-client-2025-11-20`) to connect Claude directly to your services via a local or remote MCP server:

```json
{
  "betas": ["mcp-client-2025-11-20"],
  "tools": [
    {
      "type": "mcp_client_20250820",
      "name": "your_mcp_server",
      "config": {
        "command": "node",
        "args": ["./mcp-server.js"],
        "env": {"API_KEY": "secret"}
      }
    }
  ]
}
```

The MCP server handles API calls, auth, and data transformation. Claude calls MCP tools instead of making raw requests.

**When to use:** Private APIs, authentication required, production systems, complex data pipelines.

## 4. **You Pass Data Directly**
The simplest approach: your application fetches from your API and passes the data to Claude in the conversation:

```json
{
  "messages": [
    {
      "role": "user",
      "content": "Analyze this user data: " + json.dumps(live_data)
    }
  ]
}
```

**When to use:** You control the backend, data isn't massive, you want full control over what Claude sees.

## Quick Decision Tree

| Scenario | Solution |
|----------|----------|
| Public API, no auth, testing | **Web Fetch** |
| Public API + code logic | **Code Execution** |
| Private API, self-hosted, production | **MCP Connector** |
| You control the backend | **Pass data in message** |
| Many APIs, complex workflow | **MCP + multiple servers** |

- **No direct outbound by default**: Claude Messages API cannot initiate requests.
- **Code execution limits**: Sandboxed, has egress but can't reach `localhost` or private networks.
- **MCP is the enterprise pattern**: Scales to multiple services, handles auth cleanly, no token cost for tool definitions (uses Tool Search).
