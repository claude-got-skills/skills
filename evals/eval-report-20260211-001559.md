# Document Analysis Pipeline Architecture

Here's a practical setup scaled for a small team:

## Core Components

**1. Document Processing Layer**
```
PDF Input → PDF Parser → Text Extraction → Chunking
```
- Use **PyPDF2** or **pdfplumber** for reliable extraction
- Split into logical chunks (by page or semantic boundaries) to avoid token limits
- Store raw + processed versions for debugging

**2. Information Extraction**
- Use an LLM API (OpenAI, Claude, Ollama) with structured prompts
- Define a schema for what you're extracting (dates, names, amounts, etc.)
- Chain multiple focused extraction steps rather than one massive prompt
- Example: extract facts → identify risks → cross-reference separately

**3. Knowledge Base Integration**
- Start simple: **SQLite** for <100k documents, **PostgreSQL** for growth
- Store reference data with embeddings (use sentence-transformers, it's free)
- Query via semantic search when cross-referencing

**4. Summary & Output**
- Template-based formatting (Jinja2)
- Version control summaries in git (lightweight audit trail)

## Practical Implementation


# Document Analysis Pipeline Architecture

For a small team processing 100-page PDFs, I'd recommend a **two-stage, single-model pipeline** that's simple to operate but scales efficiently.

## Recommended Architecture

```
PDF → Extraction (Sonnet) → Validation + Enrichment (Sonnet) → Summary (Sonnet)
       ↓                          ↓
    Files API              Knowledge base lookup
```

### Why This Design

**Model choice: Sonnet 4.5**
- 200K context window handles full 100-page PDFs without chunking
- Extended thinking for reasoning through complex relationships
- 2-3x cheaper than Opus; overkill for extraction/summarization
- Reserve Opus only if you hit edge cases (ambiguous relationships, conflicting data)

**Single pipeline vs. subagents**
- Your workflow is sequential and low-complexity: extract → cross-reference → summarize
- Subagents add overhead (context isolation cost, complexity) without benefit here
- Keep it in one conversation until the PDF size or reasoning depth demands isolation

**Files API, not re-uploads**
- Upload the PDF once, reuse across conversation turns
- 500MB per file, 100GB per org — plenty for your use case
- Saves tokens and bandwidth on multi-turn workflows

## Key Decisions Explained

| Decision | Rationale |
|----------|-----------|
| **200K context, not chunking** | Avoids chunking complexity. Sonnet's context is large enough that most 100-page PDFs fit whole. No information loss at boundaries. |
| **Single conversation, not batch** | Batch API (50% discount) only makes sense if latency doesn't matter. For small team iteration, streaming + interactivity wins. |
| **Effort: "medium"** | Low cost, strong reasoning. Only push to "high" if you hit extraction misses on complex relationships. |
| **Adaptive thinking** | Sonnet can use extended thinking when needed (cross-referencing logic) but won't waste tokens on simple extraction. |
| **No custom tool for KB lookup** | KB cross-reference happens in-context (pass KB summary in system prompt or user message). Simpler than tool definition, still effective for small teams. |

## When to Reconsider

You might need **subagents** if:
- PDFs routinely exceed 150 pages and start hitting context limits
- Different document types (contracts vs. financial reports) need separate extraction schemas
- The knowledge base becomes so large that context gets constrained

You might need **batch processing** if:
- You're processing 100+ PDFs weekly with no time pressure
- Cost matters more than latency

You might need **Opus** if:
- Extraction accuracy on ambiguous documents drops below acceptable threshold
- Complex multi-document cross-referencing is required

# QA System Architecture for AI-Generated Content

This is a strong use case for **multi-agent decomposition** combined with **structured outputs**. Here's the architecture:

## Core Design: Orchestrator + Specialist Subagents

```
User Content
    ↓
[Orchestrator Agent]
    ├─→ [Accuracy Checker Subagent]
    ├─→ [Tone & Voice Subagent]
    └─→ [Compliance Checker Subagent]
         ↓
    [Aggregator]
         ↓
    Structured QA Report
```

**Why this pattern:**
- Each checker has its own system prompt optimized for that domain
- Isolated context windows prevent cross-contamination (tone feedback won't bias accuracy scoring)
- Parallel execution potential (though sequential is simpler to start)
- Easy to add/remove checkers without restructuring

## Implementation Approach

### 1. **Orchestrator System Prompt**
Set clear instructions for decomposition:

```
You are a QA coordinator. Your job is to route content through specialized 
review agents and synthesize their findings into a final report.

For each piece of content:
1. Send to accuracy_checker subagent
2. Send to tone_checker subagent  
3. Send to compliance_checker subagent
4. Aggregate results into structured format

Provide each subagent with ONLY the content and their specific context 
(e.g., don't send style guide to accuracy checker).
```

### 2. **Specialized Subagents**

**Accuracy Checker:**
- System prompt includes domain-specific knowledge, fact-checking resources, or links to knowledge bases
- Tool: web search for fact verification (if needed)
- Output schema: `{issues: [{type, severity, location, suggestion}]}`

**Tone & Voice Checker:**
- System prompt includes your brand voice guidelines, tone examples
- Analyzes for consistency with established brand personality
- Output schema: `{tone_violations: [{issue, example_text, recommendation}]}`

**Compliance Checker:**
- System prompt embeds your style guide, legal/regulatory requirements
- Checks for prohibited terms, required disclaimers, accessibility standards
- Output schema: `{compliance_issues: [{rule, violation, location, fix}]}`

### 3. **Structured Output Format**

## 2. Create an API Route (app/api/chat/route.ts)

```typescript
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();

export async function POST(request: Request) {
  const { messages, context } = await request.json();

  const stream = client.messages.stream({
    model: "claude-3-5-sonnet-20241022",
    max_tokens: 1024,
    system: `You are a helpful assistant analyzing user data. ${context || ""}`,
    messages: messages,
  });

  // Convert to ReadableStream for streaming response
  const encoder = new TextEncoder();
  const readable = new ReadableStream({
    async start(controller) {
      try {
        for await (const chunk of stream) {
          if (
            chunk.type === "content_block_delta" &&
            chunk.delta.type === "text_delta"
          ) {
            controller.enqueue(
              encoder.encode(chunk.delta.text)
            );
          }
        }
        controller.close();
      } catch (error) {
        controller.error(error);
      }
    },
  });

  return new Response(readable, {
    headers: {
      "Content-Type": "text/event-stream",
      "Cache-Control": "no-cache",
      Connection: "keep-alive",
    },
  });
}
```

```typescript
  const [messages, setMessages] = useState<Array<{ role: string; content: string }>>([]);
  const [input, setInput] = useState("");
  const [loading, setLoading] = useState(false);

  const handleSubmit = async (e: React.FormEvent) => {
    e.preventDefault();
    if (!input.trim()) return;

    // Add user message

      const response = await fetch("/api/chat", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          messages: [...messages, userMessage],
          userData: "user's data here", // Pass relevant user data
        }),
      });

      if (!response.ok) throw new Error("API request failed");

      // Stream the response
      const reader = response.body?.getReader();
      if (!reader) return;

      let fullResponse = "";
      const decoder = new TextDecoder();

      while (true) {
        const { done, value } = await reader.read();
        if (done) break;

        // Parse SSE format
        const chunk = decoder.decode(value);
        const lines = chunk.split("\n");

        for (const line of lines) {
          if (line.startsWith("data: ")) {
            const data = JSON.parse(line.slice(6));
            if (data.type === "content_block_delta") {
              fullResponse += data.delta.text;
              setMessages((prev) => {
                const updated = [...prev];
                updated[updated.length - 1] = {
                  role: "assistant",
                  content: fullResponse,
                };
                return updated;
              });
            }
          }
        }
      }
    } catch (error) {
      console.error("Error:", error);
    } finally {
      setLoading(false);
    }
  };

      <div className="flex-1 overflow-y-auto space-y-2">

            {msg.content}
          </div>
```

## How They Work

**TCP (Transmission Control Protocol):**
- Establishes a connection before sending data (three-way handshake)

- Retransmits lost packets automatically
- Ensures data arrives exactly as sent, in the correct order

- Sends data immediately without setup

- No mechanism to detect or recover lost packets

- **File transfers (FTP, SFTP)** — Data integrity is critical
