# Document Analysis Pipeline Architecture

For a small team processing 100-page PDFs, I'd recommend a **straightforward two-stage pipeline** that balances simplicity with the constraints of Claude's context window. Here's what I'd build:

## Recommended Architecture

```
PDF Input → Chunking Layer → Stage 1: Extract → Stage 2: Cross-ref + Summarize → JSON Output
```

### Stage 1: Extraction (Per Document)
Use **Haiku** with the Files API to extract structured data from each PDF in parallel:
- Upload the PDF once via Files API (avoids re-uploading)
- Extract key entities, dates, amounts, relationships
- Output a JSON intermediate format (~500-2000 tokens per document)
- Cost-efficient; Haiku is built for this

### Stage 2: Cross-reference & Summarize (Consolidated)
Use **Sonnet** to:
- Take the extracted data from all documents
- Query your knowledge base (via MCP if it's external, or direct lookup if local)
- Produce a final structured summary
- This runs once per batch, not per document

### Why This Works for You

| Factor | Decision | Why |
|--------|----------|-----|
| **Model choice** | Haiku → Sonnet | Haiku handles high-volume extraction; Sonnet handles cross-reference logic that needs judgment |
| **File handling** | Files API | Upload once, reference by ID across requests. Saves bandwidth and tokens |
| **Context window** | 200K per request | Haiku fits ~50-100 extracted summaries + context; Sonnet fits extracted data + knowledge base context |
| **Cost** | Batch API for Stage 1 | 50% discount on extraction (typically your highest-volume step) |
| **Knowledge base** | MCP if remote, direct if local | If it's a database/API, MCP lets Claude query it; if it's reference data in JSON, pass it directly |

## Simple Implementation Sketch

```python

from typing import Any

- Key entities (names, organizations, amounts)
- Important dates
- Relationships between entities

Return as JSON with these fields: entities, dates, relationships, topics"""
                    },
                    {
                        "type": "document",

                        }
                    }
                ]
            }
        ]
    )
    return json.loads(response.content[0].text)

# Stage 2: Cross-reference and summarize

    extracted_data: list[dict],
    knowledge_base: dict
) -> dict:
    """Use Sonnet for intelligent cross-referencing"""
    response = client.messages.create(
        model="claude-sonnet-4-5-20250929",
        max_tokens=4000,
        messages=[
            {
                "role": "user",
                "content": f"""Cross-reference this extracted data against our knowledge base.

Extracted Data:
{json.dumps(extracted_data, indent=2)}

Knowledge Base:
{json.dumps(knowledge_base, indent=2)}

For each entity/relationship:
1. Mark if it's known (in KB) or new
2. Flag any conflicts between documents
3. Infer missing connections

Return structured summary as JSON."""
            }
        ]
    )

# Usage
```

Route strategically to minimize overhead:

- **Auto-approve**: Low-risk content passing all checks (perhaps 40-60%)
- **Sampled review**: Spot-check passing items by category
- **Full review**: Anything flagged + high-stakes content (financial, medical, legal)

```
Reviewer decision → Pattern analysis → Retrain/adjust rules
```

# QA System Architecture for AI-Generated Content

This is a multi-stage filtering problem. Here's the architecture I'd recommend, with reasoning on the key choices:

## Core Architecture: Multi-Agent Pipeline

```
Input Content
    ↓
[Extraction Agent] → Extract key claims, tone markers, style elements
    ↓
[Parallel QA Agents]
├─ Accuracy Checker
├─ Tone Validator
└─ Style Compliance
    ↓
[Aggregator Agent] → Consolidate findings, determine pass/fail
    ↓
Output (approved or flagged for human review)
```

### Why subagents instead of tool chaining?

Each QA dimension (accuracy, tone, style) needs:
- Its own system prompt with domain expertise (fact-checking vs. tone analysis are different skills)
- Different tool sets (accuracy might use web search; style uses your guide)
- Isolated context to prevent bias (one agent's findings shouldn't color another's judgment)

A single agent with sequential tools would conflate these concerns and waste tokens reasoning about irrelevant dimensions.

## Implementation Pattern

**Use subagents for each QA dimension:**

```python
from anthropic import Anthropic

client = Anthropic()

def qa_pipeline(content: str, style_guide: str) -> dict:
    # Stage 1: Extraction (fast, single agent)
    extraction = client.messages.create(
        model="claude-opus-4-5-20251101",
        max_tokens=1000,
        system="""Extract key claims, identified tone, and style elements.
Return JSON with: claims[], tone_markers[], style_elements[]""",
        messages=[{"role": "user", "content": f"Content:\n{content}"}]
    )
    
    # Stage 2: Parallel QA checks (using subagents)
    accuracy_check = accuracy_agent(content, extraction)
    tone_check = tone_agent(content, extraction)
    style_check = style_agent(content, style_guide, extraction)
    
    # Stage 3: Aggregation
    final_report = client.messages.create(
        model="claude-opus-4-5-20251101",
        max_tokens=2000,
        system="""You are a QA aggregator. Review findings and determine:
1. Overall pass/fail
2. Severity of any issues (critical/warning/info)
3. Recommended actions""",
        messages=[{
            "role": "user",
            "content": f"""
Accuracy findings: {accuracy_check}
Tone findings: {tone_check}
Style findings: {style_check}

Consolidate and recommend next steps.
"""
        }]
    )
    
    return parse_report(final_report)
```

## Key Architectural Decisions

### 1. **Model Tiering**
- **Sonnet** for extraction and individual QA checks (cost-efficient, fast)
- **Opus** for final aggregation where you need judgment about severity and policy
- Reasoning: Extraction is deterministic; QA is pattern-matching. Aggregation requires judgment about organizational risk.

### 2. **System Prompts as Domain Expertise**

Each subagent's system prompt should embed your knowledge:


```javascript
const client = new Anthropic();

export async function POST(request) {
```

```python
        [{'name': 'Alice', 'age': 30}, {'name': 'Bob', 'age': 25}]

    valid_policies = {"use_default", "skip", "raise"}
    if missing_policy not in valid_policies:
        raise ValueError(
            f"missing_policy must be one of {valid_policies}, "
            f"got '{missing_policy}'"
        )

    if missing_policy == "raise":
        # Check all keys exist first
        if not all(key in item for item in data):
```

## How They Work

**TCP (Transmission Control Protocol):**
- Establishes a connection via 3-way handshake before data flows
- Tracks every packet; retransmits lost ones
- Ensures data arrives in the correct order
- Like sending a registered letter — delivery is confirmed

**UDP (User Datagram Protocol):**
- Sends packets immediately without setup
- No acknowledgment or retransmission
- Packets may arrive out of order or get lost
- Like shouting across a room — you just hope they hear you

## When to Use Each

### Use **TCP** when:
